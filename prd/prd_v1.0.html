延迟报名需求

区域政企地推场景中，客户扫码报名后系统原会立即经统一平台传入 X 计划并触发电销致电，导致客户产生信息泄露与过度打扰的负面感知。本需求通过在统一平台对 BI 四级来源配置延迟报名时间，将报名记录传入 X 计划的动作延后，从而降低致电挂断率。

一、需求背景

区域政企合作地推场景中，线下推广物料放置区域专属企微码，客户扫码添加企微后，欢迎语推送报名链接。客户报名后，系统当前会立即将报名记录经公众号统一平台传入 X 计划，电销客服随即发起致电。

由于「报名 → 立即致电」链路过短，客户普遍产生信息泄露、被过度打扰的负面感知，整体挂断率居高不下。

二、需求目标

每个报名来源 source 支持单独设置延迟报名时间，通过报名来源 source 控制报名记录执行延迟报名。

3.1 来源记录新增「延迟设置」按钮与设置弹窗

3.1.1 前端规则逻辑

  • 位置：BI 四级来源配置列表页，「操作」列中，在「编辑」按钮右侧新增「延迟设置」按钮。
  • 弹窗标题「延迟报名设置」；弹窗上方展示当前来源名称与当前已配置的延迟时间：四级来源名称 XXXX（精确来源：XXXX）；当前延迟报名时间：NN 分钟。
  • 校验：必须为非负整数。
  • 展示映射：保存后列表「延迟报名」列按配置值 >0 显示「是」（标记延迟报名=是），0 或空表示不延迟、显示「否」。

3.3.1 前端规则逻辑

  • 位置：报名列表 → 详情弹窗，在「是否传X计划」下方新增两项展示字段——「延迟报名时间」「传入 X 计划时间」。
  • 展示映射：延迟报名 = 0 时，「延迟报名时间」显示 0（不延迟），「传入 X 计划时间」= 真实调用时间；延迟 > 0 时，定时任务传 X 计划后回填传入 X 计划时间。

3.3.2 交互逻辑

  • 报名详情打开即展示上述两字段，值由后端返回（无额外交互）。

3.4 新增延迟报名定时检查逻辑
定时检查流程
flowchart TD A[定时任务 每分钟执行] --> B[查询 延迟报名>0 且 传入X计划时间为空] B --> C{当前时间-报名时间 >= 延迟报名时间?} C -- 否 --> D[跳过 不传入] C -- 是 --> E[调用X计划报名接口] E --> F[更新 传入X计划时间=真实调用时间]
flowchart TD
  A[定时任务 每分钟执行] --> B[查询 延迟报名>0 且 传入X计划时间为空]
  B --> C{当前时间-报名时间 >= 延迟报名时间?}
  C -- 否 --> D[跳过 不传入]
  C -- 是 --> E[调用X计划报名接口]
  E --> F[更新 传入X计划时间=真实调用时间]

3.4.1 后端处理逻辑

  • 调度：每分钟执行一次。
  • 查询：取「延迟报名时间 > 0 且 传入 X 计划时间为空」的报名记录。
  • 判断：当前时间 − 报名记录报名时间 ≥ 延迟报名时间 → 触发传入。
  • 业务处理：调用 X 计划报名接口执行报名；成功后更新该记录「传入 X 计划时间」= 真实调用时间。
  • 未达延迟时间：跳过，不传入 X 计划。
